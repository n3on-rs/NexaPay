\section{Pr\'esentation de NexaPay}

NexaPay est une plateforme fintech compl\`ete con\c{c}ue pour le march\'e tunisien. Elle combine un portefeuille num\'erique, une passerelle de paiement pour commer\c{c}ants, et un moteur blockchain propri\'etaire avec consensus BFT \`a 3 validateurs.

\subsection{URLs Publiques}

    \item[\href{https://sandbox.nexapay.space}{sandbox.nexapay.space}] Plateforme principale (dashboard, envois, carte)
    \item[\href{https://auth.nexapay.space}{auth.nexapay.space}] Authentification (login, inscription)
    \item[\href{https://backend.nexapay.space}{backend.nexapay.space}] API Backend
    \item[\href{https://nexapay.space/docs}{nexapay.space/docs}] Documentation API \& SDK
\end{description}

\subsection{Architecture}

\begin{itemize}[leftmargin=*]
    \item \textbf{Frontend} --- Next.js 16 avec routage par sous-domaines
    \item \textbf{Backend} --- Rust/Axum avec JWT, cookies httpOnly, AES-256-GCM
    \item \textbf{Blockchain} --- Cha\^ine append-only, 3 validateurs BFT, signatures Ed25519
    \item \textbf{Base de donn\'ees} --- PostgreSQL 16 + SQLite + Sled
    \item \textbf{SDK} --- \texttt{@nexapay/node-sdk} sur npm
\end{itemize}

\section{S\'ecurit\'e et Protection des Donn\'ees}

La s\'ecurit\'e est au c\oe{}ur de l'architecture NexaPay. Chaque composant a \'et\'e con\c{c}u avec les meilleures pratiques de l'industrie financi\`ere.

\subsection{Protection des Donn\'ees Utilisateur}

\textbf{Donn\'ees personnelles :}
\begin{itemize}[leftmargin=*]
    \item Les donn\'ees sensibles (CIN, num\'ero de t\'el\'ephone, email) sont stock\'ees dans PostgreSQL avec acc\`es strictement contr\^ol\'e
    \item Les recherches de comptes ne retournent que les 4 derniers chiffres du t\'el\'ephone et du CIN --- jamais les donn\'ees compl\`etes
    \item Toutes les communications entre le navigateur et le serveur passent par HTTPS/TLS 1.3
\end{itemize}

\textbf{Donn\'ees bancaires :}
\begin{itemize}[leftmargin=*]
    \item Les num\'eros de carte Visa et CVV sont chiffr\'es avec AES-256-GCM (standard militaire)
    \item Seuls les 4 derniers chiffres de la carte sont visibles --- le num\'ero complet n'est jamais affich\'e
    \item Les cl\'es priv\'ees Ed25519 des utilisateurs sont chiffr\'ees avec une cl\'e d\'eriv\'ee de leur PIN personnel via Argon2id
    \item Chaque utilisateur poss\`ede un \textbf{salt al\'eatoire unique} pour le hachage de son PIN --- rendant les attaques par dictionnaire impossibles
\end{itemize}

\textbf{Sessions et Authentification :}
\begin{itemize}[leftmargin=*]
    \item Double authentification : PIN (6 chiffres) + code OTP envoy\'e par SMS
    \item Les sessions utilisent des cookies \textbf{httpOnly} (inaccessibles au JavaScript, immunis\'es contre le vol par XSS)
    \item Les tokens JWT utilisateur et administrateur sont sign\'es avec des \textbf{cl\'es cryptographiques distinctes}
    \item Les sessions peuvent \^etre r\'evoques c\^ot\'e serveur \`a tout moment
    \item Protection contre les attaques par force brute : rate limiting par adresse IP sur toutes les routes d'authentification
\end{itemize}

\textbf{Transactions et Blockchain :}
\begin{itemize}[leftmargin=*]
    \item Chaque transaction est enregistr\'ee dans un registre blockchain immuable (append-only)
    \item Les signatures Ed25519 garantissent la non-r\'epudiation --- aucune transaction ne peut \^etre falsifi\'ee
    \item Le consensus \`a 3 validateurs assure qu'aucune entit\'e seule ne peut modifier l'\'etat de la cha\^ine
    \item Les webhooks (notifications commer\c{c}ants) sont sign\'es avec HMAC-SHA256 pour garantir leur authenticit\'e
\end{itemize}

\subsection{Pourquoi 3 Validateurs (BFT) ?}

NexaPay utilise un consensus \textbf{Byzantine Fault Tolerant} \`a 3 validateurs. Contrairement aux syst\`emes centralis\'es (un seul serveur) ou aux blockchains publiques (lentes et co\^uteuses) :

\begin{itemize}[leftmargin=*]
    \item \textbf{Aucun point de d\'efaillance unique} --- Si un validateur tombe en panne, les deux autres continuent de fonctionner
    \item \textbf{Validation crois\'ee} --- Chaque bloc doit \^etre sign\'e par les 3 validateurs avant d'\^etre ajout\'e \`a la cha\^ine
    \item \textbf{Transactions instantan\'ees} --- Pas de minage, pas de proof-of-work. Les transferts sont appliqu\'es en moins d'une seconde
    \item \textbf{Co\^ut minimal} --- Pas de frais de gas, pas de commission blockchain. Frais de r\'eseau calcul\'es par tranches, visibles avant confirmation
    \item \textbf{Souverainet\'e} --- L'infrastructure est h\'eberg\'ee sur nos propres serveurs, pas sur un cloud public \'etranger
\end{itemize}

\subsection{Flux d'Authentification}

\begin{enumerate}[leftmargin=*]
    \item Utilisateur entre t\'el\'ephone + PIN (6 chiffres)
    \item V\'erification PIN avec Argon2id (sel al\'eatoire unique)
    \item Envoi OTP 6 chiffres par SMS (Twilio)
    \item V\'erification OTP en temps constant (r\'esistant aux attaques temporelles)
    \item Session \'etablie avec cookie httpOnly + JWT
\end{enumerate}

\section{Processus KYC (Know Your Customer)}

\textbf{Note importante :} Le module complet de v\'erification documentaire et KYC est d\'ej\`a d\'evelopp\'e et int\'egr\'e \`a la plateforme. Il est actuellement \textbf{d\'esactiv\'e en mode sandbox/d\'emo} pour fluidifier les tests --- les utilisateurs sont auto-v\'erifi\'es. \textbf{En production, le flux complet ci-dessous sera activ\'e.}

\subsection{V\'erification d'Identit\'e --- Particuliers}

\begin{enumerate}[leftmargin=*]
    \item \textbf{Soumission des documents}
    \begin{itemize}
        \item L'utilisateur photographie le \textbf{recto} et le \textbf{verso} de sa Carte d'Identit\'e Nationale (CIN)
        \item Upload s\'ecuris\'e avec validation de format (PNG, JPG, max 5 Mo) et stockage chiffr\'e AES-256-GCM
    \end{itemize}

    \item \textbf{Extraction automatique par OCR}
    \begin{itemize}
        \item Notre module OCR (Reconnaissance Optique de Caract\`eres) scanne les deux faces de la CIN et extrait automatiquement :\\
        --- Nom et pr\'enom complets\\
        --- Num\'ero CIN\\
        --- Date de naissance et lieu de naissance\\
        --- Date d'\'emission de la CIN
        \item Les donn\'ees extraites sont crois\'ees avec le formulaire d'inscription pour v\'erifier la coh\'erence
    \end{itemize}

    \item \textbf{Test de vivacit\'e (Liveness Detection)}
    \begin{itemize}
        \item L'utilisateur effectue un \textbf{test de vivacit\'e} via la cam\'era de son t\'el\'ephone
        \item Le syst\`eme v\'erifie qu'il s'agit d'une personne r\'eelle et vivante (pas une photo imprim\'ee, pas une vid\'eo)
        \item Mouvements demand\'es : clignement des yeux, rotation de la t\^ete, sourire
        \item Un \textbf{score de correspondance faciale} est calcul\'e entre la photo de la CIN et le selfie en direct
    \end{itemize}

    \item \textbf{V\'erification crois\'ee et approbation automatique}
    \begin{itemize}
        \item Si le score de vivacit\'e d\'epasse le seuil de confiance requis \textbf{ET} que les donn\'ees CIN correspondent au formulaire
        \item Le compte est automatiquement marqu\'e \texttt{kyc\_status = verified}
        \item \textbf{Aucune intervention humaine n\'ecessaire} --- le processus est enti\`erement automatis\'e
        \item L'utilisateur peut imm\'ediatement acc\'eder \`a toutes les fonctionnalit\'es
    \end{itemize}
\end{enumerate}

\subsection{V\'erification des Commer\c{c}ants --- Devenir Agent}

Pour les candidats agents/commer\c{c}ants, un processus de v\'erification documentaire similaire est appliqu\'e :

\begin{enumerate}[leftmargin=*]
    \item \textbf{Soumission des documents professionnels}
    \begin{itemize}
        \item \textbf{Patente / Licence commerciale} (PDF) --- scan du document officiel
        \item \textbf{Extrait du Registre National des Entreprises (RNE)} (PDF)
        \item Formulaire avec : nom commercial, type d'activit\'e, matricule fiscal, adresse, volume mensuel estim\'e
    \end{itemize}

    \item \textbf{Scan et v\'erification automatique des documents}
    \begin{itemize}
        \item Notre algorithme scanne les documents professionnels soumis
        \item Extraction automatique des informations cl\'es : raison sociale, matricule fiscal, date d'enregistrement, adresse du si\`ege
        \item V\'erification crois\'ee de la coh\'erence entre la patente et le RNE
        \item Contr\^ole de validit\'e : les documents doivent \^etre lisibles, non expir\'es, et correspondre aux informations du formulaire
    \end{itemize}

    \item \textbf{Scoring de risque automatis\'e}
    \begin{itemize}
        \item Le syst\`eme attribue un \textbf{score de risque} bas\'e sur la qualit\'e des documents, la coh\'erence des donn\'ees, et le volume estim\'e
        \item Si le score est dans la plage acceptable ET que tous les documents sont valides
        \item L'agent est \textbf{automatiquement approuv\'e} --- cr\'eation de l'espace entreprise, g\'en\'eration de la cl\'e API
        \item \textbf{Aucune intervention humaine} dans le processus standard
    \end{itemize}

    \item \textbf{Cas litigieux}
    \begin{itemize}
        \item Si le score est en dessous du seuil ou si les documents pr\'esentent des incoh\'erences
        \item Le dossier est transmis \`a un \textbf{administrateur humain} pour r\'evision manuelle
        \item L'administrateur peut approuver, rejeter, ou demander des documents compl\'ementaires
    \end{itemize}
\end{enumerate}

\subsection{Conformit\'e R\'eglementaire}

Le processus KYC et de v\'erification documentaire est con\c{c}u pour \^etre conforme avec :
\begin{itemize}[leftmargin=*]

\section{Structure des Frais --- Algorithme par Tranches}

Tous les frais NexaPay sont calcul\'es via un \textbf{algorithme par tranches} (bracket-based pricing). Contrairement \`a un pourcentage fixe qui p\'enalise les petits montants et laisse de l'argent sur la table pour les gros montants, l'algorithme par tranches optimise le revenu \`a chaque niveau de transaction.

\subsection{Algorithme de Calcul}

\[
\text{Frais}(montant) = \max\left(\text{Forfait}_{tranche},\; \text{Taux}_{tranche} \times \text{Montant} + \text{Fixe}_{tranche}\right)
\]

O\`u chaque montant tombe dans une tranche pr\'ecise. Le frais final est le maximum entre le plancher minimum et le pourcentage calcul\'e.

\subsection{Pourquoi un Algorithme par Tranches ?}

\begin{itemize}[leftmargin=*]
    \item \textbf{Micro-transactions prot\'eg\'ees} --- Un frais fixe de 0,500 TND sur 5 TND garantit que la plateforme ne perd pas d'argent sur les petits montants
    \item \textbf{Gros montants optimis\'es} --- Un taux de 1\% sur 15\,000 TND rapporte 155 TND sans para\^itre excessif
    \item \textbf{Acceptation psychologique} --- Les paliers sont con\c{c}us pour correspondre aux habitudes de consommation tunisiennes (caf\'e, d\'ejeuner, courses, loyer, v\'ehicule)
    \item \textbf{Configurable en direct} --- Les tranches sont stock\'ees en base de donn\'ees. Ajustables sans red\'eploiement de code
\end{itemize}

\subsection{Routage des Frais --- Compte Revenue On-Chain}

Chaque frais pr\'elev\'e est achemin\'e vers un \textbf{compte de revenu on-chain} d\'edi\'e :

\begin{verbatim}
Expediteur debite : montant + frais
  -> Destinataire credite : montant
  -> Compte NEXAPAY_REVENUE : frais
\end{verbatim}

Les frais ne disparaissent jamais --- ils sont enregistr\'es dans le registre blockchain immuable et enti\`erement auditables. Le solde du compte revenue peut \^etre vir\'e quotidiennement vers le compte bancaire de r\`element.

\section{Mod\`ele de Licence --- NexaPay sous Votre \'Egide}

NexaPay ne poss\`ede \textbf{aucune licence bancaire} et ne pr\'etend pas en avoir besoin. Nous op\'erons comme \textbf{front-end num\'erique} de La Banque de Tunisie, sous votre licence existante.

\subsection{Ce Que la Banque Couvre}

\begin{itemize}[leftmargin=*]
    \item \textbf{Licence bancaire} --- Toute l'activit\'e NexaPay est couverte par votre agrément BCT

\section{Phases d'Int\'egration}

L'int\'egration de NexaPay au syst\`eme de la Banque de Tunisie se fait en 4 phases progressives, con\c{c}ues pour minimiser l'effort technique c\^ot\'e banque.

\subsection{Phase 1 --- Activation Imm\'ediate (48h)}

\textbf{Aucun travail informatique requis.} Le d\'epartement commercial de la banque ouvre trois comptes group\'es :

\begin{center}
\begin{tabular}{|l|l|l|}
\hline
\textbf{Compte} & \textbf{Fonction} & \textbf{Contenu} \\
\hline
Compte 1 & R\`eglement commer\c{c}ants & Soldes group\'es des agents/commer\c{c}ants \\
Compte 2 & Revenu NexaPay & Part NexaPay des frais (revers\'e p\'eriodiquement) \\
Compte 3 & Float utilisateurs & Tous les soldes des portefeuilles utilisateurs \\
\hline
\end{tabular}
\end{center}

\textbf{D\'elai : 48 heures.} NexaPay est op\'erationnel. Les utilisateurs peuvent s'inscrire, transf\'erer, et payer.

\subsection{Phase 2 --- R\`eglement par Fichier (Semaine 1--2)}

La banque fournit un acc\`es SFTP \`a son syst\`eme de traitement par lots existant. NexaPay g\'en\`ere des fichiers de r\`eglement quotidiens au format CSV/MT940 :

\begin{verbatim}
RIB;Montant;Reference;Type
99000654364483768727;1250.500;INV-2026-001;payout_merchant
99000654364483768728;340.000;WTH-2026-045;withdrawal_user
\end{verbatim}

La banque traite ces fichiers via son syst\`eme batch existant --- celui qui g\`ere d\'ej\`a les paies, les virements utilitaires, etc. \textbf{Z\'ero d\'eveloppement n\'ecessaire c\^ot\'e banque.}

\textbf{D\'elai : 1--2 semaines} (cr\'eation acc\`es SFTP + test fichier).

\subsection{Phase 3 --- EBICS / ISO 20022 (Mois 1--3)}

Connexion via le protocole \textbf{EBICS} (Electronic Banking Internet Communication Standard), que toute banque tunisienne op\'erant des comptes corporate supporte d\'ej\`a.

\begin{itemize}[leftmargin=*]
    \item \textbf{EBICS Client} --- NexaPay impl\'emente le client EBICS pour pousser les ordres de paiement directement dans le core banking
    \item \textbf{ISO 20022 pain.001} --- Messages de paiement standardis\'es pour les virements
    \item \textbf{ISO 20022 camt.053} --- Relev\'es bancaires \'electroniques pour le rapprochement automatique
    \item \textbf{NexaPay Bank Adapter} --- Module Rust qui traduit les transactions on-chain en messages ISO 20022
\end{itemize}

\textbf{Travail banque :} Activer EBICS sur les comptes group\'es (fonctionnalit\'e existante pour les clients corporate). \\
\textbf{Travail NexaPay :} Impl\'ementation du client EBICS + module ISO 20022 ($\sim$300 lignes de Rust).

\textbf{D\'elai : 2--4 semaines.}

\subsection{Phase 4 --- Int\'egration Profonde (Mois 3--6)}

Si le volume le justifie, connexion directe au core banking pour la validation en temps r\'eel :

\begin{itemize}[leftmargin=*]
    \item \textbf{Vue DB en lecture seule} --- Validation RIB et v\'erification de solde en temps r\'eel
    \item \textbf{KYC automatis\'e} --- NexaPay soumet les donn\'ees CIN, la banque valide contre sa base de donn\'ees clients
    \item \textbf{Provisioning IBAN automatis\'e} --- Cr\'eation de sous-comptes ou IBAN virtuels \`a la vol\'ee
    \item \textbf{R\'econciliation temps r\'eel} --- Rapprochement automatique entre le ledger NexaPay et le core banking
\end{itemize}

\textbf{Travail banque :} Cr\'eation d'une vue DB read-only + whitelist IP ($\sim$1--2 jours DBA).

\subsection{R\'ecapitulatif --- Effort Banque par Phase}

\begin{center}
\begin{tabular}{|l|c|c|c|}
\hline
\textbf{Phase} & \textbf{D\'elai} & \textbf{Effort Banque} & \textbf{Qui fait quoi} \\
\hline
1 -- Comptes    & 48h      & Ouvrir 3 comptes                    & D\'epartement commercial \\

3 -- EBICS      & 2--4 sem & Activer EBICS (existant)           & IT + Corporate Banking \\

\hline
\textbf{Total}  & \textbf{3--4 mois} & \textbf{$\sim$4 jours de travail effectif} & \\
\hline
\end{tabular}
\end{center}

\textbf{Le point cl\'e :} ce n'est pas la banque qui s'int\`egre \`a NexaPay --- c'est NexaPay qui s'int\`egre \`a la banque. Nous adaptons notre plateforme \`a l'infrastructure existante de la banque, et non l'inverse.

\section{Administration et Supervision}

    \item \textbf{Solde syst\`eme} --- Vue globale des fonds en circulation

\subsection{Gestion des Utilisateurs}

\begin{itemize}[leftmargin=*]
    \item \textbf{Liste compl\`ete des comptes} --- Recherche par nom, t\'el\'ephone, CIN, ou adresse blockchain
    \item \textbf{Fiche d\'etaill\'ee par utilisateur} --- Informations personnelles, solde, historique des transactions, statut KYC, documents soumis
    \item \textbf{Gel de compte} --- L'administrateur peut \textbf{geler un compte} imm\'ediatement en cas d'activit\'e suspecte. Le compte gel\'e ne peut plus effectuer de transactions
    \item \textbf{D\'egel de compte} --- Apr\`es v\'erification manuelle, l'administrateur peut r\'eactiver le compte
    \item \textbf{Journal d'audit} --- Toute action administrative est enregistr\'ee avec horodatage, identit\'e de l'administrateur, et raison fournie
\end{itemize}

\subsection{Surveillance des Transactions}

\begin{itemize}[leftmargin=*]
    \item \textbf{Historique complet} --- Toutes les transactions de la plateforme sont visibles avec : exp\'editeur, destinataire, montant, horodatage, type, statut
    \item \textbf{Filtrage} --- Par date, statut, montant, type de transaction
    \item \textbf{Gel de transaction} --- L'administrateur peut \textbf{geler une transaction sp\'ecifique} pour v\'erification avant qu'elle ne soit d\'efinitivement confirm\'ee
    \item \textbf{Tra\c{c}abilit\'e blockchain} --- Chaque transaction a un hash on-chain v\'erifiable ind\'ependamment
\end{itemize}

\subsection{Gestion des Agents}

\begin{itemize}[leftmargin=*]
    \item \textbf{Candidatures en attente} --- Liste des demandes d'agents avec score de risque
    \item \textbf{Approbation / Rejet manuel} --- Pour les cas litigieux, l'administrateur peut approuver ou rejeter avec un commentaire
    \item \textbf{Limites de volume} --- Ajustement du plafond mensuel par agent
    \item \textbf{Suspension d'agent} --- D\'esactivation temporaire d'un compte agent
\end{itemize}

\subsection{S\'ecurit\'e Administrative}

\begin{itemize}[leftmargin=*]
    \item \textbf{Authentification renforc\'ee} --- Login administrateur avec mot de passe + TOTP (code temporaire \`a 6 chiffres, SHA-256)
    \item \textbf{Cl\'e JWT s\'epar\'ee} --- Les tokens administrateur utilisent une cl\'e cryptographique distincte des tokens utilisateur

    \item \textbf{Protection anti-seed} --- La cr\'eation du premier compte admin n\'ecessite une cl\'e secr\`ete (\texttt{NEXAPAY\_ADMIN\_SEED\_KEY})
